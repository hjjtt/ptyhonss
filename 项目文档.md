## 1. 项目概述

本项目是一个**端到端的数据采集与分析系统**，围绕"动漫用户长评"这一主题，完整覆盖了从数据抓取、存储、导出到可视化分析的全流程。

- **数据源**：MyAnimeList（全球最大的动漫评分社区），通过其公开 REST API **Jikan** 获取
- **数据规模**：13000+ 条真实长评，覆盖 200+ 部热门动漫、9000+ 独立用户
- **核心能力**：爬虫（含限速/重试/增量缓存）+ SQLite 数据库 + Excel 导出 + Numpy/Pandas/Matplotlib 7 张分析图
- **全部真实数据**：无任何模拟或伪造，所有评论均来自真实用户在 MyAnimeList 上的公开发言

## 2. 项目背景与目标

### 背景
课程要求实现一个满足以下条件的数据项目：
1. 真实爬取 10000+ 条结构化数据
2. 存入数据库（字段齐全）
3. 使用 Numpy + Pandas + Matplotlib 生成至少 2 张分析图
4. 提交 Excel、SQL 表结构、源码等成果物

### 设计思路
将"动漫作品"类比为"商家"，"用户长评"类比为"用户评价"，从而把动漫评论数据完美映射到通用的"评论/评价分析"业务场景中。

### 为什么选 Jikan
| 候选数据源 | 选用与否 | 原因 |
|---|---|---|
| GitHub Issues API | ✗ | 需 Token，评论短、偏技术 |
| 携程/电商评论 | ✗ | 反爬强、风控严 |
| **Jikan (MyAnimeList)** | ✓ | 公开免鉴权、长评丰富（数百字）、字段齐全 |

## 6. 模块详解

### 6.1 `scraper/` 爬虫层

#### `config.py` — 配置中心
集中管理所有可调参数，避免魔法数字散落代码：

| 参数 | 值 | 说明 |
|---|---|---|
| `JIKAN_API` | `https://api.jikan.moe/v4` | API 根地址 |
| `REQUEST_INTERVAL` | 1.5 秒 | 请求间隔（远低于 Jikan 3 req/s 上限） |
| `REQUEST_TIMEOUT` | 15 秒 | 单请求超时 |
| `MAX_RETRIES` | 5 | 失败重试次数 |
| `MAX_ANIME` | 80 | 每轮最多新增动漫数 |
| `MAX_REVIEWS_PER_ANIME` | 100 | 单动漫 API 翻页上限（5 页 × 20） |
| `MAX_REVIEWS_PER_ANIME_LIMIT` | 1000 | **课程规则**: 单动漫评论数硬上限 |
| `RECENT_YEARS` | 2 | **课程规则**: 仅保留近 N 年评论 |
| `ENABLE_FILTER` | `True` | **总开关**: False 则抓全量历史 |
| `ANIME_QUERIES` | `[("bypopularity","tv",80)]` | 抓取策略 |
| `RATING_RANGES` | 见下文 | 1-10 → 1-5 归一化规则 |

评分归一化规则：
```
MAL score ≥ 9  → 5 星
MAL score ≥ 7  → 4 星
MAL score ≥ 5  → 3 星
MAL score ≥ 3  → 2 星
其他            → 1 星
```

#### `http_client.py` — HTTP 客户端
封装 `requests.Session`，提供三个核心能力：

- **节流（throttle）**：两次请求间至少间隔 `REQUEST_INTERVAL` 秒，用时间戳差值计算等待
- **随机 UA 轮换**：内置 4 套 User-Agent（iPhone/Android），每次请求随机选一套
- **指数退避重试**：失败时按 `2^attempt × 0.5` 秒退避；遇 429（rate limit）单独等更久

对外只暴露 `get(url, params)`，成功返回 `Response`，失败返回 `None`（业务层优雅降级）。

#### `crawler.py` — 抓取核心
两层抓取流程：

```
crawl_all()
   │
   ├── 1. fetch_anime_list()      取动漫列表 (翻页, 25/页)
   │       GET /top/anime?filter=bypopularity&type=tv
   │
   └── 2. _fetch_anime_reviews()  逐个动漫抓评论 (5 页 × 20)
           GET /anime/{mal_id}/reviews
           │
           ├── 过滤: _is_recent() 只保留近两年
           └── _parse_review()    解析单条 → Comment 对象
```

**数据过滤规则**（课程要求，默认开启，`ENABLE_FILTER=True`）：
- **单动漫上限**：`min(MAX_REVIEWS_PER_ANIME, MAX_REVIEWS_PER_ANIME_LIMIT)` = min(100, 1000) = 100 条
- **时间窗口**：每条评论的 `date` 与 `_cutoff_date()`（当前时刻往前推 `RECENT_YEARS` 年）比较，早于截止日的直接丢弃
- 关闭规则：`ENABLE_FILTER = False`，则抓全量历史数据

**数据模型**用 `@dataclass` 定义：

- `Comment`：comment_id / user_id / user_name / merchant_name / rating / comment_time / comment_content / like_count
- `AnimeInfo`：mal_id / title / 各语言标题 / type / episodes / score / rank / popularity / year / image_url

抓取完成后 `_dump_raw_json()` 把当轮结果按时间戳落盘，作为永久缓存。

#### `cache.py` — 增量缓存
`load_all_raw()` 扫描 `data/raw/` 下所有 JSON：
- 动漫按 `mal_id` 去重（同 id 取最新）
- 评论按 `comment_id` 去重

这样即使多轮抓取、Jikan 接口变动，也能从本地缓存完整还原数据。

### 6.2 `database/` 存储层

#### `schema.sql` — 表结构（幂等）
使用 `CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`，重复执行不破坏数据。视图 `v_comment_full` 用 `DROP VIEW IF EXISTS` + `CREATE` 保证定义最新。

#### `db_manager.py` — CRUD 封装
`DBManager` 类，通过上下文管理器管理连接：

```python
@contextmanager
def _conn(self):
    conn = sqlite3.connect(self.db_path)
    conn.row_factory = sqlite3.Row    # 行按字典式访问
    try:
        yield conn
        conn.commit()                 # 正常提交
    except Exception:
        conn.rollback()               # 异常回滚
        raise
    finally:
        conn.close()
```

主要方法：
- `upsert_anime(anime_list)` — `INSERT OR REPLACE` 批量写入动漫元数据
- `insert_comments(comments)` — `INSERT OR IGNORE` 按 comment_id 去重插入评论
- `stats()` — 返回总数/动漫数/独立用户数/平均评分

#### `exporter.py` — Excel 导出
读 `v_comment_full` 视图，导出含 3 个 sheet 的 `data.xlsx`：

| Sheet | 内容 |
|---|---|
| 全部评论 | 13033 条完整评论（13 列，评论ID 在首列） |
| 动漫列表 | 203 部动漫元数据，按评分人数降序 |
| 按动漫汇总 | 每部动漫的评论数/平均评分/总 Reactions |

### 6.3 `analysis/` 分析层

#### `visualizer.py` — 7 张图

## 9. 数据流与运行流程

`main.py::run_pipeline()` 编排 5 个步骤：

```
[启动]
   │
   ▼
Step 0: 创建目录 (data/, output/, logs/)
   │
   ▼
Step 1: 初始化数据库
   DBManager() 读 schema.sql 建表 + 索引 + 视图
   │
   ▼
Step 2: 抓取 / 加载
   ├─ skip_crawl=False:
   │    2a. load_all_raw() 加载已有 raw 缓存
   │    2b. crawl_all() 抓新动漫 (跳过已抓的 mal_id)
   │    2c. load_all_raw() 重新加载完整数据 (新+旧)
   │    2d. upsert_anime() + insert_comments() 写库
   │
   └─ skip_crawl=True:
        load_all_raw() 直接从缓存加载 → 写库
   │
   ▼
Step 3: 统计 (db.stats()) + 校验 ≥10000
   │
   ▼
Step 4: 导出 data.xlsx (export_to_excel)
   │    3 个 sheet: 全部评论 / 动漫列表 / 按动漫汇总
   │    复制一份为 data.xlxs (课程要求的错名)
   ▼
Step 5: 分析出图 (generate_all_charts)
   load_dataframe() → Numpy 统计 → 7 张图
   │
   ▼
[完成, 耗时约 20-30 秒]
```

## 11. 运行方式

### 环境准备

```bash
# 1. 安装依赖 (5 个包, 无需任何服务端)
pip install -r requirements.txt
```

### 三种运行模式

```bash
# 模式 A: 完整流程 (抓取 + 入库 + 导出 + 分析)
python main.py

# 模式 B: 跳过抓取, 用 raw 缓存 + 已有数据库分析 (推荐复跑)
python main.py --skip-crawl
```

### 增量抓取机制

抓取是**累积式**的：
- 每次运行只抓 `data/raw/` 中不存在的新动漫
- 原始 JSON 永久保存，按时间戳命名（`jikan_<时间戳>.json`）
- 下次运行自动跳过已抓动漫，只补充新数据
- 即使 Jikan 接口变动，本地缓存仍可完整还原全部数据

## 12. 工程特性

### 抗限速设计
- **固定间隔节流**：1.5 秒/请求，远低于 Jikan 3 req/s 上限
- **指数退避重试**：失败按 `2^attempt × 0.5` 秒退避，最多 5 次
- **429 单独处理**：遇 rate limit 等待更久
- **随机 User-Agent**：4 套 UA 轮换，降低被识别概率

### 数据可靠性
- **原始 JSON 落盘**：每轮抓取结果永久保存到 `data/raw/`
- **去重机制**：raw 层按 `comment_id` 去重，DB 层用 `INSERT OR IGNORE`
- **事务管理**：DBManager 用上下文管理器，异常自动回滚
- **幂等建表**：`schema.sql` 可重复执行不破坏数据

### 日志体系
- 同时输出到控制台和 `logs/run.log`
- 格式：`[时间] 级别 模块: 消息`
- 关键节点均有日志（每步、每动漫、每张图）

## 14. 已知限制与扩展方向

### 当前限制
1. **抓取受限于 Jikan 3 req/s**：全量抓取需 5-10 分钟（含节流）
2. **`comment_content` 截断 1500 字**：超长评论会被裁剪
3. **评分维度偏乐观**：MAL 长评用户偏向打高分（实测均值 4.089/5）
4. **分词为简单 2-gram**：未用 jieba，中文分词粒度有限

### 可扩展方向
- **数据库升级**：抽象 `_conn()` 即可切换 MySQL/PostgreSQL，支持多人协作
- **增量调度**：配合 cron/任务调度实现每日自动抓取
- **更深度分析**：接入情感分析（snownlp / transformers）做正负面挖掘
- **Web 可视化**：用 Flask/Streamlit 把 7 张图做成交互式 Dashboard
- **推荐系统**：基于用户-动漫评分矩阵做协同过滤推荐
